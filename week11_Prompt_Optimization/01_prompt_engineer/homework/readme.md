## ข้อ 1: 🏥 ระบบวิเคราะห์อาการเบื้องต้นและให้คำแนะนำสุขภาพ

### 📋 โจทย์

สร้างระบบที่รับ **คำอธิบายอาการป่วยเป็นภาษาไทย** แล้วทำสิ่งต่อไปนี้:

1. **จำแนกระดับความเร่งด่วน** เป็น 3 ระดับ: `ฉุกเฉิน`, `ควรพบแพทย์เร็ว`, `ดูแลตัวเองได้`
2. **วิเคราะห์สาเหตุที่เป็นไปได้** พร้อมเหตุผลทีละขั้นตอน
3. **ให้คำแนะนำเบื้องต้น** ที่เหมาะสมกับอาการ

### 🛠️ เทคนิคที่ต้องใช้ (ขั้นต่ำ)

| ขั้นตอน | เทคนิค | เหตุผลที่ต้องใช้ |
|---------|--------|-----------------|
| จำแนกระดับความเร่งด่วน | **Few-shot Prompting** | ให้ตัวอย่างอาการ + ระดับความเร่งด่วน เพื่อให้ LLM จำแนกได้ตรงรูปแบบ |
| วิเคราะห์สาเหตุ | **Chain-of-Thought** | ให้ LLM อธิบายเหตุผลทีละขั้นตอน ไม่ใช่ตอบลอยๆ |
| ตรวจสอบความมั่นใจ | **Self-Consistency CoT** | รันหลายครั้ง แล้วดูว่าคำตอบสอดคล้องกันหรือไม่ |

### 📥 Input ตัวอย่าง (ใช้ทดสอบทั้ง 3 case)

```
Case 1: "ปวดหัวข้างเดียวมา 2 วัน คลื่นไส้ แพ้แสง"
Case 2: "เจ็บคอ มีไข้ต่ำๆ เป็นมา 1 วัน"
Case 3: "แน่นหน้าอก หายใจลำบาก เหงื่อออก ปวดร้าวไปแขนซ้าย"
```

### 📤 Output ที่คาดหวัง (ตัวอย่างโครงสร้าง)

สำหรับแต่ละ case ต้องแสดง:

```
🔴 ระดับความเร่งด่วน: [ฉุกเฉิน / ควรพบแพทย์เร็ว / ดูแลตัวเองได้]

🔍 การวิเคราะห์สาเหตุ (Chain-of-Thought):
   ขั้นตอน 1: ...
   ขั้นตอน 2: ...
   ขั้นตอน 3: ...

📊 ความสอดคล้อง (Self-Consistency):
   - ผลรัน 5 ครั้ง: [แสดงผลแต่ละรอบ]
   - Consistency Score: xx%

💊 คำแนะนำเบื้องต้น: ...
```

### 📊 เกณฑ์การประเมิน (Evaluation)

| เกณฑ์ | รายละเอียด |
|-------|-----------|
| Few-shot Design | ตัวอย่างครอบคลุม 3 ระดับ, หลากหลาย, format ชัดเจน |
| CoT Quality | ขั้นตอนการคิดมีเหตุมีผล สมเหตุสมผลทางการแพทย์เบื้องต้น |
| Self-Consistency | รันอย่างน้อย 5 ครั้ง, มี consistency score, วิเคราะห์ผลได้ |
| Code Quality | โค้ดมี function ชัดเจน, มี helper function, รันได้ไม่ error |
| Evaluation (ดูรายละเอียดด้านล่าง) | มีการประเมินผลลัพธ์ด้วยวิธีการที่เรียนจาก Lab อย่างน้อย 2 วิธี |

### 📊 Evaluation ที่ต้องทำ

นักศึกษาต้องเพิ่ม **ส่วนประเมินผลลัพธ์ (Evaluation Section)** โดยใช้เทคนิคที่เรียนจาก Lab Part 9-10 **อย่างน้อย 2 วิธี** จากตัวเลือกต่อไปนี้:

| วิธี Evaluation | คำอธิบาย | ใช้ตอนไหน |
|----------------|----------|----------|
| **LLM-as-a-Judge** | เขียน evaluation prompt ให้ LLM ตรวจคุณภาพคำตอบ โดยใช้ rubric ที่กำหนดเกณฑ์ชัดเจน ให้คะแนนแต่ละด้านพร้อมเหตุผล | ประเมินคุณภาพคำตอบภาพรวม |
| **Reference-free Evaluation** | ประเมินคุณภาพโดยไม่ต้องมีคำตอบอ้างอิง ดู fluency, coherence, informativeness, factual consistency | ประเมินคำแนะนำสุขภาพที่ไม่มี ground truth |
| **Hallucination Detection** | ตรวจสอบว่าผลลัพธ์มีข้อมูลที่แต่งขึ้นหรือไม่ ตรวจทุก factual claim | ตรวจสอบความน่าเชื่อถือทางการแพทย์ |
| **Pairwise Comparison (Debiased)** | เปรียบเทียบผลลัพธ์ 2 แบบ พร้อมสลับตำแหน่งเพื่อลด position bias | เปรียบเทียบผลจากเทคนิคต่างกัน เช่น CoT vs Few-shot เดี่ยว |
| **Multi-Agent Debate** | ใช้หลาย persona (เช่น Technical Expert, Communication Specialist, Devil's Advocate) ประเมินจากมุมมองต่างกัน | ประเมินคำแนะนำจากหลายมุมมอง |
| **Rubric-based Scoring** | กำหนด rubric เฉพาะทาง เช่น ความสมเหตุสมผลทางการแพทย์, ความปลอดภัยของคำแนะนำ | ประเมินตามเกณฑ์เฉพาะของโดเมนสุขภาพ |

**ตัวอย่าง Evaluation Output ที่คาดหวัง:**

```
═══════════════════════════════════════
📊 Evaluation Section
═══════════════════════════════════════

⚖️ วิธีที่ 1: LLM-as-a-Judge
   - Relevance: x/10 — [เหตุผล]
   - Accuracy: x/10 — [เหตุผล]
   - Completeness: x/10 — [เหตุผล]
   - Safety: x/10 — [เหตุผล]
   - Overall: x/10
   - Strengths: ...
   - Weaknesses: ...

🔎 วิธีที่ 2: Hallucination Detection
   - Total claims found: N
   - Verified: N
   - Uncertain: N
   - Likely False: N
   - Hallucination Risk: [LOW/MEDIUM/HIGH]
   - Overall Trustworthiness: x/10
```

### 💡 คำแนะนำ

- ตัวอย่างใน Few-shot ต้อง **ครบทั้ง 3 ระดับ** (ฉุกเฉิน, ควรพบแพทย์, ดูแลเอง) ถ้าขาดระดับใดไป LLM จะจำแนกได้ไม่ดี
- ใน CoT ให้กำหนดโครงสร้างการคิดชัดเจน เช่น "1) ระบุอาการหลัก 2) เชื่อมโยงอาการ 3) ประเมินความเสี่ยง 4) สรุป" จะดีกว่าแค่บอกว่า "คิดทีละขั้นตอน"
- Self-Consistency ต้องตั้ง `temperature > 0` (แนะนำ 0.5-0.7) ถ้า temperature = 0 ผลจะเหมือนกันทุกรอบ ไม่มีประโยชน์
- **⚠️ ย้ำ:** ระบบนี้เป็นแบบฝึกหัดเท่านั้น ไม่ใช่คำแนะนำทางการแพทย์จริง

## ข้อ 2: 📊 ระบบสร้างแผนธุรกิจอัตโนมัติ

### 📋 โจทย์

สร้างระบบที่รับ **ข้อมูลธุรกิจเบื้องต้น** แล้วสร้าง **แผนธุรกิจฉบับย่อ** ผ่านกระบวนการหลายขั้นตอน โดยต้องผ่านการ **ตรวจสอบและปรับปรุง** ก่อนเป็นผลลัพธ์สุดท้าย

### 🛠️ เทคนิคที่ต้องใช้ (ขั้นต่ำ)

| ขั้นตอน | เทคนิค | สิ่งที่ต้องทำ |
|---------|--------|-------------|
| ขั้นที่ 1 | **Prompt Chaining** | แบ่งเป็น 4 chain: วิเคราะห์ตลาด → SWOT → กลยุทธ์ → แผนปฏิบัติ |
| ขั้นที่ 2 | **Tree-of-Thought** | สร้าง 3 ทางเลือกกลยุทธ์ → ประเมินแต่ละทาง → เลือกทางที่ดีที่สุด |
| ขั้นที่ 3 | **Reflection** | ทบทวนแผนทั้งหมด → หาจุดอ่อน → ปรับปรุง |
| ขั้นที่ 4 | **Self-Critique** | วิพากษ์แผนอย่างเข้มงวด → แก้ไขจุดที่ถูกวิพากษ์ |

### 📥 Input ตัวอย่าง

```
ชื่อธุรกิจ: ร้านขนมไทยออนไลน์ "ขนมย่าสม"
ประเภท: อาหาร / ขนมไทยโบราณ
กลุ่มเป้าหมาย: คนรุ่นใหม่ อายุ 20-35 ปี ที่สนใจขนมไทย
งบประมาณเริ่มต้น: 100,000 บาท
จุดเด่น: สูตรขนมไทยดั้งเดิมจากคุณย่า + packaging สมัยใหม่
ปัญหา: ยังไม่มี brand awareness, ไม่มีหน้าร้าน
```

> 📌 นักศึกษาสามารถเปลี่ยนเป็นธุรกิจที่ตนเองสนใจได้

### 📤 Output ที่คาดหวัง

ระบบต้องแสดงผลลัพธ์ของ **ทุกขั้นตอน** ที่ผ่านมา:

```
═══════════════════════════════════════
📌 Chain 1: การวิเคราะห์ตลาด
═══════════════════════════════════════
[ผลลัพธ์]

═══════════════════════════════════════
📌 Chain 2: SWOT Analysis
═══════════════════════════════════════
[ผลลัพธ์]

═══════════════════════════════════════
📌 Chain 3: กลยุทธ์ (Tree-of-Thought)
═══════════════════════════════════════
  🌳 ทางเลือก 1: ... (คะแนน: x/10)
  🌳 ทางเลือก 2: ... (คะแนน: x/10)
  🌳 ทางเลือก 3: ... (คะแนน: x/10)
  ✅ เลือก: ทางเลือกที่ ... เพราะ ...

═══════════════════════════════════════
📌 Chain 4: แผนปฏิบัติ
═══════════════════════════════════════
[ผลลัพธ์]

═══════════════════════════════════════
🔍 Reflection: ทบทวนแผน
═══════════════════════════════════════
  จุดแข็ง: ...
  จุดอ่อน: ...
  สิ่งที่ปรับปรุง: ...

═══════════════════════════════════════
⚔️ Self-Critique: วิพากษ์แผน
═══════════════════════════════════════
  [ผลการวิพากษ์]

═══════════════════════════════════════
🏆 แผนธุรกิจฉบับสุดท้าย
═══════════════════════════════════════
[แผนที่ผ่านการปรับปรุงทั้งหมดแล้ว]
```

### 📊 เกณฑ์การประเมิน (Evaluation)

| เกณฑ์ | รายละเอียด |
|-------|-----------|
| Prompt Chaining (4 chains) | output ของ chain ก่อนหน้าถูกส่งเป็น input ของ chain ถัดไปจริง |
| Tree-of-Thought | มี 3 ทางเลือกที่แตกต่างกัน, มีคะแนน, มีเหตุผลในการเลือก |
| Reflection Quality | วิเคราะห์จุดอ่อนได้จริง ไม่ใช่แค่ชม + แผนที่ปรับปรุงดีขึ้นจริง |
| Self-Critique Quality | วิพากษ์อย่างเข้มงวด หาข้อผิดพลาด/สมมติฐานที่ไม่มีหลักฐาน |
| Code & Presentation | โค้ดเป็นระเบียบ มี function แยกชัด แสดงผลอ่านง่าย |
| Evaluation (ดูรายละเอียดด้านล่าง) | มีการประเมินแผนธุรกิจด้วยวิธีการที่เรียนจาก Lab อย่างน้อย 2 วิธี |

### 📊 Evaluation ที่ต้องทำ

นักศึกษาต้องเพิ่ม **ส่วนประเมินแผนธุรกิจฉบับสุดท้าย (Evaluation Section)** โดยใช้เทคนิคที่เรียนจาก Lab Part 9-10 **อย่างน้อย 2 วิธี** จากตัวเลือกต่อไปนี้:

| วิธี Evaluation | คำอธิบาย | ใช้ตอนไหน |
|----------------|----------|----------|
| **LLM-as-a-Judge (Rubric-based)** | ใช้ rubric ประเมินแผนสุดท้าย อย่างน้อย 4 เกณฑ์ เช่น feasibility, completeness, practicality, innovation, risk_management พร้อมคะแนนและเหตุผล | ประเมินคุณภาพแผนธุรกิจภาพรวม |
| **Hallucination Detection** | ตรวจสอบว่าแผนมีข้อมูลที่แต่งขึ้นหรือไม่ เช่น สถิติตลาด, ตัวเลขที่อ้างอิงโดยไม่มีแหล่งที่มา, สมมติฐานที่ไม่สมเหตุสมผล | ตรวจสอบความน่าเชื่อถือของข้อมูลในแผน |
| **Pairwise Comparison (Debiased)** | เปรียบเทียบแผนก่อนและหลัง Reflection/Self-Critique พร้อมสลับตำแหน่งเพื่อลด position bias | วัดว่ากระบวนการปรับปรุงทำให้แผนดีขึ้นจริงหรือไม่ |
| **Reference-free Evaluation** | ประเมินคุณภาพแผนโดยไม่ต้องมีแผนอ้างอิง ดู coherence, informativeness, factual consistency | ประเมินคุณภาพแผนแบบ open-ended |
| **Multi-Agent Debate** | ใช้หลาย persona ประเมิน เช่น Business Strategist, Financial Analyst, Devil's Advocate | ประเมินแผนจากหลายมุมมองทางธุรกิจ |
| **Semantic Similarity** | เปรียบเทียบความสอดคล้องระหว่าง SWOT Analysis กับกลยุทธ์ที่เลือก | ตรวจสอบว่ากลยุทธ์สอดคล้องกับการวิเคราะห์จริงหรือไม่ |
| **Automated Metrics (BLEU/ROUGE)** | วัดความสอดคล้องระหว่าง chain outputs ถ้ามี reference plan | เปรียบเทียบกับแผนอ้างอิง (ถ้ามี) |
